## 1. Что такое Go и чем он отличается

Go (или Golang) — язык от Google, созданный в 2009 году. Его главные особенности:

**Компилируемый** — Go компилируется в бинарный файл. Чтобы запустить приложение на сервере, не нужен интерпретатор (как Python) или виртуальная машина (как Java). Один файл — и всё работает.

**Статически типизированный** — каждая переменная имеет конкретный тип, и компилятор проверяет правильность типов до запуска. Это ловит целый класс ошибок ещё на этапе написания кода.

**Без исключений** — в Go нет `try/catch`. Вместо этого функции возвращают ошибку явно:

```go
result, err := someFunction()
if err != nil {
    // обработать ошибку
}
```

Это кажется многословным, но делает код предсказуемым — ты всегда видишь, где может пойти что-то не так.

**Горутины и конкурентность** — Go создан для многопоточной работы. `go someFunction()` запускает функцию в отдельном потоке (горутине) без лишних усилий.

**Нет наследования классов** — Go не объектно-ориентированный в классическом смысле. Вместо классов — структуры (`struct`), вместо интерфейсов с явной реализацией — утиная типизация.

## 2. Как устроен Go-проект (модули и пакеты)

### go.mod — паспорт проекта

Файл `go.mod` — это как `package.json` в Node.js или `requirements.txt` в Python. Он говорит:

- как называется наш модуль (`link-storage-service`)
- какая версия Go используется
- какие внешние зависимости нужны

```
module link-storage-service   ← имя модуля (используется в импортах)

go 1.22                        ← минимальная версия Go

require github.com/lib/pq v1.10.9  ← единственная внешняя зависимость
```

`github.com/lib/pq` — это драйвер для PostgreSQL. Он нужен, чтобы стандартная библиотека Go умела разговаривать с Postgres.

### go.sum — контрольные суммы зависимостей

`go.sum` генерируется автоматически командой `go mod tidy`. Он хранит хэши всех зависимостей — это гарантия, что при сборке на другой машине скачается ровно тот же код.

### Пакеты — организация кода

В Go код делится на **пакеты** (packages). Каждая папка — это отдельный пакет. Первая строка каждого файла объявляет, какому пакету он принадлежит:

```go
package domain   // этот файл — часть пакета domain
```

Импорт другого пакета:

```go
import "link-storage-service/internal/domain"
```

**Правило видимости:** если имя начинается с заглавной буквы — оно публичное (доступно из других пакетов). Если со строчной — приватное.

```go
type Link struct { ... }    // публичный тип — виден снаружи
type entry struct { ... }   // приватный тип — только внутри пакета
```

### Папка internal/

`internal/` — это специальное соглашение Go. Код внутри этой папки **нельзя импортировать из других проектов**. Это защита: если кто-то захочет использовать наш код как библиотеку, он не сможет залезть во внутренности.

## 3. Чистая архитектура — зачем и как

### Проблема, которую она решает

Представь, что ты написал весь код в одном файле. Сначала удобно. Потом ты захочешь сменить PostgreSQL на MySQL — и придётся переписывать половину приложения, потому что логика перемешана с работой с базой.

**Чистая архитектура** говорит: разделяй код на слои, где каждый слой знает только о том, что ниже него.

```
handler  →  service  →  repository  →  database
   ↑            ↑             ↑
HTTP-запросы  Логика      SQL-запросы
```

- **handler** знает про HTTP, но не знает про SQL
- **service** знает про бизнес-правила, но не знает ни про HTTP, ни про SQL
- **repository** знает про SQL, но не знает про HTTP

Если завтра ты решишь заменить HTTP на gRPC — менять нужно только слой handler. Остальное не трогаешь.

### Как это реализовано через интерфейсы

В Go **интерфейс** — это набор методов. Любой тип, у которого есть эти методы, автоматически реализует интерфейс. Явно объявлять `implements Interface` не нужно.

В `service/link_service.go` мы объявляем:

```go
type LinkRepository interface {
    Create(ctx context.Context, link *domain.Link) error
    GetByShortCode(ctx context.Context, shortCode string) (*domain.Link, error)
    // ...
}
```

Сервис работает с этим **интерфейсом**, а не с конкретной реализацией. Это значит:

- можно подменить PostgreSQL на любую другую базу
- можно подменить реальную базу на фейковую в тестах

## 4. Слой domain — модель данных

```
internal/domain/link.go
```

Это **сердце** приложения. Здесь описаны основные сущности и ошибки. Этот пакет **ни от чего не зависит** — он не импортирует ни базу данных, ни HTTP, ничего.

```go
type Link struct {
    ID          int64
    ShortCode   string
    OriginalURL string
    CreatedAt   time.Time
    Visits      int64
}
```

**`json:"short_code"`** — это теги. Они говорят пакету `encoding/json`, как называть поля при сериализации в JSON. Без тега поле `ShortCode` в JSON выглядело бы как `"ShortCode"` (с заглавной буквы).

```go
var (
    ErrNotFound = errors.New("link not found")
    ErrConflict = errors.New("short code already exists")
)
```

Ошибки объявлены как **переменные-сентинелы**. Это позволяет сравнивать их через `errors.Is(err, domain.ErrNotFound)` в любом месте программы — не сравнивая строки, а сравнивая сами объекты ошибок.

## 5. Слой config — переменные окружения

```
internal/config/config.go
```

Конфигурация через переменные окружения — стандарт для серверных приложений (12-factor app). Так можно запустить одно и то же приложение на локальной машине, в тестовой среде и в продакшене — просто задав разные переменные, не меняя код.

```go
type Config struct {
    HTTPAddr    string
    DatabaseURL string
    CacheTTL    time.Duration
    CacheSweep  time.Duration
}
```

`time.Duration` — это тип Go для промежутков времени. `60 * time.Second` — 60 секунд. Удобно тем, что нельзя случайно перепутать секунды с миллисекундами.

```go
func getEnv(key, defaultVal string) string {
    if v := os.Getenv(key); v != "" {
        return v
    }
    return defaultVal
}
```

`os.Getenv` читает переменную окружения. Если переменная не задана — используем значение по умолчанию. Это делает локальный запуск удобным: не нужно задавать все переменные вручную.

## 6. Слой repository — работа с базой данных

```
internal/repository/postgres/link_repo.go
```

### database/sql — стандартный интерфейс к базам

В Go работа с базами данных идёт через пакет `database/sql` из стандартной библиотеки. Сам пакет не знает о конкретной базе — он только определяет интерфейс. Конкретная реализация подключается через **драйвер**:

```go
import _ "github.com/lib/pq"
```

Знак `_` означает: импортируй пакет только ради его **побочного эффекта** — при импорте `lib/pq` регистрирует себя как драйвер для Postgres. После этого `sql.Open("postgres", ...)` знает, как подключиться.

### Параметризованные запросы

```go
r.db.QueryRowContext(ctx,
    `SELECT ... FROM links WHERE short_code = $1`,
    shortCode,
)
```

`$1` — это плейсхолдер. Значение подставляется библиотекой отдельно от текста запроса. Это **критически важно** для безопасности: без этого была бы возможна SQL-инъекция (когда пользователь передаёт в поле значение вроде `'; DROP TABLE links; --`).

### Context — управление временем жизни запроса

Первый аргумент почти каждой функции — `ctx context.Context`. Контекст — это механизм отмены операций. Если пользователь закрыл HTTP-соединение до того, как ответ был отправлен, контекст отменяется, и SQL-запрос прерывается. Без этого база данных продолжала бы выполнять запросы для уже ушедших клиентов.

### IncrementVisits — атомарное обновление

```go
`UPDATE links SET visits = visits + 1 WHERE short_code = $1`
```

Почему не "прочитать visits, прибавить 1, сохранить"? Потому что если два запроса придут одновременно, оба прочитают `visits = 10`, оба запишут `visits = 11`, и один переход потеряется. SQL-оператор `visits + 1` выполняется **атомарно** внутри базы — конкурентные запросы не мешают друг другу.

## 7. Кэш — зачем и как работает

```
internal/repository/cache/memory_cache.go
```

### Зачем нужен кэш

Каждый запрос к базе данных — это сетевой вызов (даже если база на той же машине). При большой нагрузке это становится узким местом. Кэш хранит результаты в памяти самого процесса — обращение к нему в тысячи раз быстрее.

### TTL (Time To Live)

Данные в кэше устаревают: если кто-то удалил ссылку, кэш об этом не знает. TTL — время жизни записи. После истечения TTL запись считается невалидной и при следующем запросе данные снова берутся из базы.

```go
type entry struct {
    link      *domain.Link
    expiresAt time.Time     // когда запись устаревает
}
```

### sync.RWMutex — конкурентная безопасность

Наш сервер обрабатывает много запросов одновременно (в разных горутинах). Если две горутины одновременно пишут в одну и ту же map — это **гонка данных** (data race), которая может привести к панике или повреждению данных.

`sync.RWMutex` решает это:

- `RLock()` / `RUnlock()` — блокировка на чтение. Несколько горутин могут читать одновременно.
- `Lock()` / `Unlock()` — эксклюзивная блокировка на запись. Пока одна горутина пишет, никто другой не может ни читать, ни писать.

```go
func (c *MemoryCache) Get(shortCode string) (*domain.Link, bool) {
    c.mu.RLock()         // захватить блокировку чтения
    e, ok := c.data[shortCode]
    c.mu.RUnlock()       // освободить
    // ...
}
```

### Фоновая горутина очистки

```go
go c.cleanupLoop()
```

`go` перед вызовом функции запускает её как горутину — в отдельном потоке исполнения. `cleanupLoop` периодически проходит по всей карте и удаляет устаревшие записи, чтобы память не росла бесконечно.

`time.NewTicker(c.sweep)` создаёт тикер — канал, в который каждые `sweep` секунд отправляется сигнал. `for range ticker.C` ждёт этих сигналов бесконечно.

## 8. Слой service — бизнес-логика

```
internal/service/link_service.go
```

Здесь живёт **логика приложения** — то, что не является ни HTTP, ни SQL. Сервис принимает зависимости через конструктор:

```go
func NewLinkService(repo LinkRepository, cache LinkCache) *LinkService {
    return &LinkService{repo: repo, cache: cache}
}
```

Это называется **Dependency Injection** (внедрение зависимостей). Сервис не создаёт репозиторий сам — он получает его снаружи. Это позволяет в тестах передать фейковый репозиторий, который ничего не пишет в базу.

### Генерация короткого кода

```go
func generateShortCode() (string, error) {
    b := make([]byte, 6)
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    for i, v := range b {
        b[i] = charset[int(v)%len(charset)]
    }
    return string(b), nil
}
```

`crypto/rand.Read` заполняет срез байт **криптографически случайными** числами (от ОС). Затем каждый байт берётся по модулю длины алфавита, чтобы получить символ из `charset`. Результат — 6 случайных символов вроде `aB3kR9`.

Почему `crypto/rand`, а не `math/rand`? `math/rand` — псевдослучайный генератор, его результаты можно предсказать, зная зерно. `crypto/rand` — настоящая случайность.

### Стратегия кэширования (Cache-Aside)

```go
func (s *LinkService) Get(ctx context.Context, shortCode string) (*domain.Link, error) {
    // 1. Проверить кэш
    if cached, ok := s.cache.Get(shortCode); ok {
        s.repo.IncrementVisits(ctx, shortCode)
        result := *cached    // копируем, чтобы не изменять кэшированный объект
        result.Visits++
        s.cache.Delete(shortCode)  // инвалидируем — следующий запрос возьмёт свежие данные
        return &result, nil
    }

    // 2. Промах кэша — идём в базу
    link, err := s.repo.GetByShortCode(ctx, shortCode)
    // ...
    s.cache.Set(shortCode, link)  // кладём в кэш для следующих запросов
    return link, nil
}
```

Паттерн называется **Cache-Aside** (кэш рядом): приложение само управляет кэшем — сначала смотрит туда, при промахе идёт в базу и заполняет кэш.

`result := *cached` — звёздочка разыменовывает указатель и **копирует структуру**. Без этого мы бы изменяли объект, который хранится в кэше, что могло бы привести к неожиданному поведению при конкурентных запросах.

## 9. Слой handler — HTTP API

```
internal/handler/link_handler.go
```

### net/http — стандартная библиотека

В Go не нужен внешний HTTP-фреймворк (как Express в Node.js). Стандартный пакет `net/http` вполне справляется. С Go 1.22 `ServeMux` поддерживает роутинг с методами и параметрами пути:

```go
mux.HandleFunc("GET /links/{short_code}", h.get)
```

`{short_code}` — это параметр пути, который извлекается так:

```go
shortCode := r.PathValue("short_code")
```

### Обработка запроса

```go
func (h *LinkHandler) create(w http.ResponseWriter, r *http.Request) {
    var req struct {
        URL string `json:"url"`
    }
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil || req.URL == "" {
        writeError(w, http.StatusBadRequest, "url is required")
        return
    }
    // ...
}
```

- `w http.ResponseWriter` — через него пишем ответ
- `r *http.Request` — в нём запрос (тело, заголовки, параметры)
- `json.NewDecoder(r.Body).Decode(&req)` — десериализует JSON из тела запроса в структуру
- После `writeError` обязательно `return` — без него функция продолжит выполнение

### Почему интерфейс, а не конкретный тип

```go
type linkService interface {
    Create(ctx context.Context, originalURL string) (*domain.Link, error)
    // ...
}
```

Handler знает о сервисе через **интерфейс**, а не через `*service.LinkService`. Это даёт возможность в будущем написать тесты, где вместо реального сервиса передаётся заглушка.

### Маппинг ошибок в HTTP-коды

```go
func writeErr(w http.ResponseWriter, err error) {
    if errors.Is(err, domain.ErrNotFound) {
        writeError(w, http.StatusNotFound, "link not found")
        return
    }
    writeError(w, http.StatusInternalServerError, err.Error())
}
```

`errors.Is` проверяет, является ли ошибка (или любая в её цепочке) нужным типом. Это позволяет корректно конвертировать доменные ошибки в HTTP-статусы: `ErrNotFound` → 404, всё остальное → 500.

## 10. main.go — точка сборки

```
cmd/api/main.go
```

`main.go` — единственное место, где все части собираются вместе. Именно здесь создаются конкретные реализации и передаются зависимости:

```go
func main() {
    cfg := config.Load()          // 1. Загрузить конфигурацию

    db, _ := sql.Open(...)        // 2. Подключиться к базе данных
    migrate(db)                   // 3. Создать таблицы если нет

    repo := postgres.NewLinkRepository(db)          // 4. Создать репозиторий
    memCache := cache.NewMemoryCache(...)           // 5. Создать кэш
    svc := service.NewLinkService(repo, memCache)  // 6. Создать сервис (передать зависимости)
    h := handler.NewLinkHandler(svc)               // 7. Создать хендлер

    mux := http.NewServeMux()     // 8. Создать роутер
    h.RegisterRoutes(mux)         // 9. Зарегистрировать маршруты

    http.ListenAndServe(addr, mux) // 10. Запустить сервер
}
```

Это называется **Composition Root** — единое место, где конфигурируется граф зависимостей. Весь остальной код не знает, какие конкретные реализации используются.

### Миграции при старте

```go
func migrate(db *sql.DB) error {
    _, err := db.Exec(`CREATE TABLE IF NOT EXISTS links (...)`)
    return err
}
```

`IF NOT EXISTS` означает: создай таблицу, если её ещё нет. При повторном запуске это не упадёт с ошибкой. Для продакшн-проектов обычно используют специальные инструменты миграций (golang-migrate), но для учебного проекта это вполне нормально.

## 11. Docker — как упаковать приложение

### Dockerfile — рецепт сборки образа

```dockerfile
FROM golang:1.22-alpine AS builder   # Этап 1: используем Go для сборки

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download                   # Скачать зависимости (кэшируется)
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/api  # Скомпилировать

FROM alpine:3.19                      # Этап 2: минимальный образ для запуска
WORKDIR /app
COPY --from=builder /app/server .    # Скопировать только бинарник
CMD ["./server"]
```

Это **многоэтапная сборка** (multi-stage build). Образ Go весит ~300 МБ — это много для финального контейнера. Мы собираем бинарник в большом образе, а затем копируем только его в маленький `alpine` (~5 МБ). Финальный образ весит ~10 МБ.

`CGO_ENABLED=0` — отключить CGO (C-интеграция). Тогда бинарник будет статически слинкован — не зависит ни от каких системных библиотек и запустится в любом Linux-контейнере.

### docker-compose.yml — два контейнера

```yaml
services:
  app: # наш сервис
    build: .
    depends_on:
      db:
        condition: service_healthy # ждать пока база не готова

  db: # PostgreSQL
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"] # проверка готовности
```

`condition: service_healthy` — приложение не запустится, пока база данных не пройдёт healthcheck. Без этого приложение попытается подключиться до того, как PostgreSQL инициализирован, и упадёт.

## 12. Как это всё связано вместе

Проследим путь одного запроса `POST /links`:

```
1. Клиент отправляет HTTP-запрос
   POST /links
   {"url": "https://example.com"}

2. net/http принимает запрос, находит обработчик по паттерну "POST /links"

3. handler.create() вызывается
   - читает JSON из тела запроса
   - вызывает svc.Create(ctx, "https://example.com")

4. service.Create() выполняет бизнес-логику
   - генерирует случайный short_code, например "aB3kR9"
   - создаёт объект domain.Link
   - вызывает repo.Create(ctx, link)

5. postgres.Create() выполняет SQL
   - INSERT INTO links (short_code, original_url, ...) VALUES (...)
   - заполняет link.ID возвращённым значением

6. Управление возвращается в handler
   - отправляет JSON-ответ клиенту:
     {"short_code": "aB3kR9"}
```

Каждый слой делает ровно одно дело и не знает о деталях соседей. Это и есть чистая архитектура в действии.

## Итого: ключевые концепции

| Концепция                 | Где используется      | Зачем                                                |
| ------------------------- | --------------------- | ---------------------------------------------------- |
| Интерфейсы                | service, handler      | Развязать слои, упростить тесты                      |
| Dependency Injection      | main.go, конструкторы | Не создавать зависимости внутри, получать снаружи    |
| Context                   | все слои              | Отмена запросов, дедлайны                            |
| sync.RWMutex              | cache                 | Безопасность при конкурентном доступе                |
| Параметризованные запросы | repository            | Защита от SQL-инъекций                               |
| Sentinel errors           | domain                | Явная обработка конкретных ошибок                    |
| Multi-stage Docker        | Dockerfile            | Минимальный финальный образ                          |
| Cache-Aside               | service.Get           | Кэшировать при промахе, инвалидировать при изменении |
